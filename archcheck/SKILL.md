# /archcheck 技能

## 定位

`/archcheck` 是一个**架构体检工具**,基于以下三本经典:

- Robert C. Martin《架构整洁之道》(Clean Architecture, 2017)
- Alistair Cockburn《Hexagonal Architecture》(Ports & Adapters, 2005)
- Eric Evans《领域驱动设计》(DDD, 2003)

它不是泛泛地说"你的代码不好",而是**用具体的、可验证的标准**告诉你:

- 业务核心被框架污染了多少行
- 哪些接口的归属错了
- 删掉 adapter 包后业务能不能独立编译
- 改一个需求要扩散到多少个文件

## 何时触发

- 用户输入 `/archcheck <项目路径>` 或 `/archcheck` 后跟代码
- 用户说"帮我做架构体检"、"分析这个项目"、"看看符不符合整洁架构"、"评估这个项目"
- 用户传上来一个代码包/zip/目录,说"看看怎么样"

## 五大体检维度

```
┌─────────────────────────────────────────────────────┐
│  维度 1:依赖方向 (Dependency Direction)             │
│  → 业务核心是否被框架污染                            │
│  → 接口归属是否正确(端口在 domain 还是 adapter)     │
│  → 业务核心能否独立编译                              │
├─────────────────────────────────────────────────────┤
│  维度 2:边界完整性 (Boundary Integrity)             │
│  → Web 边界、持久化边界、外部系统边界、模块边界       │
├─────────────────────────────────────────────────────┤
│  维度 3:Entity vs UseCase 区分                      │
│  → 充血还是贫血、业务规则归属、Use Case 是否纯编排    │
├─────────────────────────────────────────────────────┤
│  维度 4:可测试性 (Testability)                      │
│  → 业务核心能否脱离框架测试、mock 密度、测试外部依赖  │
├─────────────────────────────────────────────────────┤
│  维度 5:代码坏味道 (Code Smells)                    │
│  → SOLID 违反、巨型类、命名失真、DTO/Domain 混淆     │
└─────────────────────────────────────────────────────┘
```

## 执行流程

### Phase 0:工程识别

收到 `/archcheck` 命令后,先做**工程识别**:

1. **识别技术栈**
   - 看 `pom.xml` / `build.gradle` → Java/Spring 项目
   - 看 `go.mod` → Go 项目
   - 看 `package.json` → TS/JS 项目
   - 看 `pyproject.toml` / `requirements.txt` → Python 项目

2. **识别项目规模**
   - 统计源码文件数、行数(只统计业务代码,排除测试和生成代码)
   - 识别主要包/模块结构

3. **识别架构风格(初判)**
   - 找有没有 `domain` / `application` / `usecase` / `adapter` / `port` 这类目录
   - 找有没有 `controller` / `service` / `repository` 这种典型分层
   - 形成**初步假设**:这是传统三层?DDD?整洁架构?

4. **加载对应的语言模式库**
   - Java/Spring 项目 → 必读 `references/language-patterns/java-spring.md`
   - 其他语言 → 读对应文件

### Phase 1:并行扫描五大维度

按维度依次读取参考文件,对项目执行扫描:

| 维度 | 必读参考 |
|------|---------|
| 依赖方向 | `references/dependency-direction.md` |
| 边界完整性 | `references/boundary-integrity.md` |
| Entity vs UseCase | `references/entity-vs-usecase.md` |
| 可测试性 | `references/testability.md` |
| 代码坏味道 | `references/code-smells.md` |

每个维度的输出格式:

```
## 维度 X:[名称]

### 评分:A/B/C/D/F

### ✓ 做对了的
- [具体观察 + 文件:行号]

### ✗ 扣分项
- 【严重度】观察到的现象
  - 文件:`src/main/java/.../XxxService.java:42`
  - 违反原则:DIP / OCP / 依赖朝外 / ...
  - 三层剖析:
    - 现象层:看到什么
    - 机制层:为什么这样写出问题
    - 原理层:违反了哪条架构原则
  - 改造建议:改成什么样

### 🔧 改造建议(按 ROI 排序)
- [优先级] 建议项 + 预期收益 + 改动成本
```

### Phase 2:综合评分

读取 `references/scoring-rubric.md`,综合五维分数,给出总评。

总评分级:

- **A 级(整洁架构标杆)**:依赖方向严格,业务核心可独立编译,可独立测试。罕见。
- **B 级(接近整洁)**:架构师有意识,但局部有漏洞。优秀团队的常见水平。
- **C 级(传统三层)**:Spring 应用的主流水平,能用,但变化时成本高。
- **D 级(架构有问题)**:能维护,但改一处崩三处,新人上手慢。
- **F 级(架构腐烂)**:已经成为业务发展的瓶颈,需要紧急止血。

### Phase 3:Top 5 优先改造点

跨维度归纳出 Top 5 最值得做的改造,每条标注:

- 改动成本(小/中/大)
- 预期收益(具体说能解决什么痛点)
- 推荐时机(立刻/下次重构/择机)

### Phase 4:重构路线图

给一条 3 阶段路线图:

```
Stage 1 止血 (1-2 周)
  → 不动现有代码,通过新增模块/接口先把出血点控制住
  
Stage 2 修复 (1-3 月)
  → 逐步把核心业务从框架里剥离

Stage 3 优化 (持续)
  → 引入 ArchUnit 等工具守护架构,防止退化
```

### Phase 5:输出双格式报告

**两份报告同时输出**:

1. **Markdown 报告** `archcheck-report.md`
   - 完整内容,纯文本,易于版本控制和 PR 评论

2. **HTML 可视化报告** `archcheck-report.html`
   - 基于 `assets/report-template.html` 模板
   - 深色科技风(`#0a0c10` 背景),延续 deeparch 风格
   - 五维雷达图(SVG 绘制)
   - 扣分项可折叠展开
   - 三层剖析卡片化呈现
   - 代码片段语法高亮

最终用 `present_files` 把两份报告交付给用户,**HTML 报告排在前面**(用户直接打开看)。

## 迭代支持

用户读完报告后通常会追问。处理方式:

- **追问"为什么这一项扣分"** → 深入展开三层剖析,把代码逐行分析,合并回报告
- **追问"怎么改造"** → 给出"改之前→改之后"的完整代码对照,作为附录补充
- **质疑评分** → 不要立即妥协,先把判断依据再讲一遍;如果用户提供了新信息(比如"这个类其实是 legacy 不打算改"),则更新评分并标注"已知例外"

## 报告生成规则(关键)

### 必须做的

- 每个扣分项必须 **cite 具体文件路径 + 行号**(没有就不要乱写)
- 每个扣分项必须给出**改造建议**,不能只批评
- 评分必须有**依据**,不能凭感觉给字母

### 不能做的

- 不要扣"它没用 DDD"这种主观分(DDD 是选择,不是强制)
- 不要把"代码风格问题"塞进架构维度(那是 lint 的事)
- 不要把项目和"理想整洁架构"对比就一通批,要考虑**项目阶段和团队规模**
- 不要因为用了 Spring 就扣分(用 Spring 不等于反整洁)

### 鼓励做的

- 先指出做对的部分(✓),再说扣分项(✗)
- 站在团队当下处境给建议,不要只讲"应该"
- 标注"已知工程权衡":比如 `@Transactional` 放 Service 层,严格说违反整洁架构,但实战中是合理妥协

## 与其他 skill 的关系

- 与 `/deeparch-md` 互补:`/archcheck` 关注**架构合规性**,`/deeparch-md` 关注**单点深度剖析**
- 与 `/primer` 互补:`/primer` 是入门,`/archcheck` 是评审
- 报告中如果发现某个组件值得深入,可以建议用户用 `/deeparch-md` 进一步分析该组件

## 输出语言

中文为主,代码注释和扣分项标签可双语并列(便于团队评审使用)。
