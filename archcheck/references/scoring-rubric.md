# 综合评分细则 (Scoring Rubric)

> 给项目打分不是给一个数字,而是给一个**有依据、有维度、有上下文**的判断。这一节定义评分规则。

## 五维总评映射

每个维度给字母分(A/B/C/D/F),五维加权求总评。

权重建议(可根据项目调整):

| 维度 | 权重 | 理由 |
|------|------|------|
| 1. 依赖方向 | 30% | 最硬,其他维度的基础 |
| 2. 边界完整性 | 25% | 架构本质 |
| 3. Entity vs UseCase | 15% | 内部细分,可妥协 |
| 4. 可测试性 | 20% | 反向验证前几维 |
| 5. 代码坏味道 | 10% | 影响面广但可逐步修复 |

## 字母分到数值

```
A = 4.0
B = 3.0
C = 2.0
D = 1.0
F = 0.0
```

## 总评分级

```
加权得分 ≥ 3.5  →  A (整洁架构标杆)
加权得分 ≥ 2.5  →  B (接近整洁,有意识)
加权得分 ≥ 1.5  →  C (传统三层,主流水平)
加权得分 ≥ 0.5  →  D (架构有问题,改一处崩三处)
加权得分 < 0.5  →  F (架构腐烂,需紧急止血)
```

## 各级总评的画像

### A 级:整洁架构标杆

- domain 包零框架 import
- 接口归属 100% 正确
- 业务核心可独立编译
- 95%+ 业务测试不启动 Spring/容器
- 有 ArchUnit 等架构守护测试
- 充血模型,Use Case 纯编排
- 团队对 Uncle Bob/Cockburn 的理解到位

**典型评语**:
> 这个项目代表了端口适配器架构在企业级 Java 项目里的最佳实践。可以作为团队内训练材料。少数可改进点是工程细节而非架构问题。

### B 级:接近整洁

- 有明确的 domain / application / adapter 分层
- 接口归属基本正确(>90%)
- 业务核心**大部分**可独立编译,有 1-2 处轻度污染
- 充血模型,但少量业务规则泄漏到 Service
- 测试金字塔基本健康
- 团队有架构意识,但没做到极致

**典型评语**:
> 项目架构整体健康,是少数有架构意识的团队的产物。建议从 X、Y 两处轻度污染下手,可在 1-2 周内提升到 A 级。重点是建立 ArchUnit 守护防止退化。

### C 级:传统三层(主流)

- 典型 Controller-Service-Repository 三层
- 业务逻辑在 Service 类里,Service 较厚
- Domain 类是 POJO + ORM 实体合体
- Repository 接口归属错误(在 dao/mapper 包里)
- 测试一半依赖 Spring,一半纯 JUnit
- 少量明显坏味道(巨型类、命名失真)

**典型评语**:
> 这是 Spring 项目的主流架构水平,能用、能维护、能扩展,但变化时成本较高。**不是必须重构**,而是看团队下一步目标:如果项目还会快速迭代 5 年以上,投入提升到 B 级是有 ROI 的;如果项目稳定运营,可以接受现状,把精力放在新模块上做对。

### D 级:架构有问题

- Controller 直调 Repository,跳过 Service
- Service 类巨型(2000+ 行)
- domain 包不存在,或里面是 JPA 实体
- 大量 `@Autowired` 字段注入
- 模块间循环依赖
- 测试稀少,且都是集成测试
- 业务逻辑分散,改一处需要扩散到多处

**典型评语**:
> 项目当前已经在为新需求付出明显的"架构税"。建议**不要立刻大重构**(高风险),而是采用绞杀者模式(Strangler Pattern):新功能用整洁架构方式开发,老代码维持现状,逐步替换。给出 3 阶段路线图见下文。

### F 级:架构腐烂

- 没有分层概念,所有代码混在一个包
- 一个文件几千行
- 全局静态变量、上帝类
- 严重循环依赖
- 几乎没有测试
- 改一行代码不知道会影响什么

**典型评语**:
> 项目已经成为业务发展的瓶颈。强烈建议:1) 立即停止在现有代码上加复杂功能;2) 围绕老系统做 Anti-Corruption Layer(防腐层),新功能在新模块中开发;3) 启动重写计划,但**不是从零重写**,而是逐步绞杀。

## 评分修正因素

打分时考虑以下情境调整,**不要机械应用标准**:

### 项目阶段修正

| 阶段 | 修正 |
|------|------|
| MVP / POC(< 6 月) | 不应苛求 A 级,B 级即优秀,C 级可接受 |
| 增长期(6 月-3 年) | C 级可接受,但应有"提升路线图" |
| 成熟期(3 年+) | 应至少 B 级,否则将持续付出"架构税" |
| 维护期(基本不改) | 现状即可,投入应该谨慎 |

### 团队规模修正

| 规模 | 修正 |
|------|------|
| 1-3 人 | 简单结构最重要,过度抽象反而扣分 |
| 5-15 人 | 应有清晰边界,B 级是合理目标 |
| 30+ 人 | 必须有 A/B 级架构,否则协作崩溃 |

### 业务复杂度修正

| 复杂度 | 修正 |
|---------|------|
| 简单 CRUD | C 级足矣,过度设计反而是问题 |
| 中等业务规则 | B 级是合理目标 |
| 复杂业务(电商、金融、保险) | 应该 A 级,否则业务知识无处沉淀 |
